What is STDD?

Spec & Test-Driven Development (STDD) is a three-phase methodology designed for the age of LLM-assisted coding. It enforces a strict gate between thinking about what to build and actually building it.

Where SDD — popularized by tools like Kiro, Spec-kit and Tessl and analyzed by Birgitta Böckeler (Thoughtworks, 2025) — asks you to write specs before code, and TDD — Kent Beck's discipline of Red→Green→Refactor — asks you to write tests before code, STDD fuses both into a single ordered pipeline where each phase artifact enforces the next:

The Core Principle A specification answers "what should this do?". Tests verify that answer against observable behavior. Code is just the mechanism that makes tests pass. When an LLM generates code, it must satisfy both layers of truth — not just produce plausible-looking output.

The methodology was conceived to address a specific failure mode of AI-assisted development: LLMs generate confident-looking code that passes a visual review but silently breaks existing contracts, ignores edge cases, or drifts from the original intent. STDD installs two checkpoints — spec validation and test validation — before any code ever reaches your codebase.

"The spec is the contract. The tests are the proof. The code is just one possible implementation." — STDD Design Principle

The DNA of STDD

STDD was not invented from scratch. It stands on the shoulders of two mature methodologies — absorbing what each does well and addressing what each leaves unsolved. Understanding the parent disciplines is essential to understanding why STDD is structured the way it is.

Parent · 1999

🔴 Test-Driven Development

Introduced by Kent Beck as part of Extreme Programming, TDD enforces a simple, powerful loop: write a failing test, write the minimum code to pass it, then refactor. The discipline prevents over-engineering and ensures every line of production code has a reason to exist.

The Red → Green → Refactor Cycle:

🔴
Red
Write a failing test
→
🟢
Green
Minimum code to pass
→
🔵
Refactor
Clean without breaking

What TDD gives us:

  • Verifiable "done": tests pass = feature complete
  • Regression safety: full suite must stay green
  • Design feedback: hard-to-test code = bad design
  • Confidence to refactor: tests catch regressions immediately
  • Documentation: tests describe behavior in executable form

What TDD leaves unsolved:

  • Where do the tests come from? (Usually: developer's memory)
  • Are the tests covering the right requirements?
  • Is the thing being built the right thing?
Parent · 2024–2025

📋 Spec-Driven Development

Emerging from the AI-assisted coding wave, SDD — analyzed in depth by Birgitta Böckeler (Thoughtworks) — establishes that specifications must precede code. Tools like Kiro, Spec-kit, and Tessl formalize this as a workflow with structured artifacts.

SDD Maturity Levels (Böckeler, 2025):

Level 1

Spec-First

Spec precedes code, both discarded after delivery

Level 2 ← STDD

Spec-Anchored

Spec persists as a living artifact through evolution

Level 3

Spec-as-Source

Only specs are edited; code is fully generated (Tessl)

What SDD gives us:

  • Human-readable intent: what and why, not how
  • Prevents scope drift: LLM is anchored to the spec
  • Shared language: product + engineering alignment artifact
  • Traceability: every requirement has an origin

SDD's known weaknesses (per Böckeler):

  • No executable verification — specs can be ignored by LLMs
  • Review overhead: too many markdown files, tedious to audit
  • "False sense of control" — agents still drift from specs
  • Size-of-problem mismatch: overkill for small fixes
  • Spec-as-source risks recapitulating MDD failures
Synthesis · STDD

⚡ How STDD Resolves Both Sets of Weaknesses

TDD weaknesses → STDD solutions

  • "Where do tests come from?"
    → From the spec. Tests are derived directly from acceptance criteria, not from the developer's memory or assumptions.
  • "Are we testing the right thing?"
    → Yes, because tests must be traceable 1:1 to a spec criterion. No criterion, no test.
  • "Is this the right thing to build?"
    → The spec review gate, which must be human-approved before tests are written, answers this explicitly.

SDD weaknesses → STDD solutions

  • "Specs can be ignored by LLMs"
    → Tests are the spec in executable form. LLMs cannot ignore a failing test suite.
  • "Review overhead / too many files"
    → One spec + one test file = two artifacts per feature. Tests are already the review mechanism.
  • "Agents still drift from specs"
    → Drift is caught by the test suite. Code that drifts from the spec will fail tests derived from that spec.
The synthesis insight TDD's executable verification + SDD's human-readable intent = a closed loop where the spec and the tests are mutually enforcing. The LLM cannot satisfy one without satisfying the other. This is what neither methodology achieves alone.

The Three Phases

Each phase is a hard gate. You cannot proceed to the next phase until the current one is complete and reviewed.

Phase 01 — Specification

📋 Write the Spec

Before a single line of code or test is written, the team produces a structured specification document. This is a human-readable, machine-parseable description of what the feature must do — not how.

The spec must define: goals, actors, inputs, outputs, acceptance criteria, constraints, and known edge cases. It should be detailed enough that two engineers with no prior context would reach the same implementation decisions.

  • Format: Markdown, YAML, or structured prose
  • Owner: Human (engineer + product) — LLM assists, does not author
  • Gate: Spec review / approval before moving to Phase 2
  • Output artifact: spec.md (versioned in source control)
Phase 02 — Tests

🧪 Write the Tests (from the spec)

With the spec approved, tests are written directly from its acceptance criteria. This is the moment the LLM becomes a powerful collaborator: given the spec, it can generate a comprehensive test suite — unit tests, integration tests, edge-case tests, and regression guards.

All generated tests must be reviewed by a human. Tests that cannot be derived directly from the spec are discarded. No tests may reference implementation details.

  • Source of truth: Only the spec — no guessing about implementation
  • LLM role: Generates test scaffolding from spec criteria
  • Gate: Test suite must compile, run, and all fail (red phase)
  • Output artifact: feature.test.ts (or equivalent)
Why tests must fail at this stage If a test passes before any implementation exists, it either tests the wrong thing or validates pre-existing behavior. Both are dangerous assumptions. All tests must be red before Phase 3 begins.
Phase 03 — Implementation

⚙️ Write the Code (to make tests pass)

Only after the spec is approved and the failing test suite exists does implementation begin. The LLM now has a precise target: make all tests pass without breaking any existing tests in the codebase.

The implementation goal is explicit and verifiable. There is no ambiguity about "done" — the feature is complete when the full test suite is green, including the pre-existing regression suite.

  • LLM role: Generates implementation guided by tests
  • Constraint: Must not modify test files to make tests pass
  • Gate: All new tests green + zero regressions in existing suite
  • Output artifact: Production code + updated CHANGELOG

The Feedback Loop

STDD is not strictly linear. Each phase may trigger a return to an earlier one:

Spec
→
Tests
→
Code
→
Review
↖ Test exposes spec ambiguity → revise spec  |  ↖ Code exposes impossible test → revisit spec + tests

The cycle terminates only when all three artifacts are mutually consistent.

STDD with LLMs

STDD was designed with LLM-assisted development as a first-class citizen. The methodology deliberately assigns the LLM specific, bounded roles at each phase — preventing the most common failure modes of AI coding tools.

The golden rule of STDD + LLMs The LLM reacts to what the human produces — it never originates decisions. In Phase 1 it reacts to your draft spec. In Phase 2 it reacts to the approved spec. In Phase 3 it reacts to the failing tests. Human judgment is always the source.

  1. Human drafts the spec — even if incomplete

    Start writing. Even a rough draft is enough to unlock the LLM's best contributions. The human defines goals, actors, inputs, outputs, and acceptance criteria. It does not need to be perfect — Phase 1 is iterative.

  2. LLM reviews and strengthens the spec (assistant, not author)

    This is where the LLM adds enormous value in Phase 1 — not by writing the spec, but by pressure-testing it. Use these prompts in sequence:

    Prompts for spec review 
    # 1. Find ambiguities
"Review this spec and point out any acceptance criteria that are
vague, contradictory, or that two engineers could interpret differently."

# 2. Find missing edge cases
"What edge cases, error states, or boundary conditions does this
spec not cover? List each one with a brief explanation of the risk."

# 3. Check testability
"Is this spec detailed enough to generate tests without making
any assumptions? List every assumption a developer would have to make."

# 4. Socratic questioning
"Ask me 5 questions about this spec whose answers would change
the implementation significantly."

# 5. Final completeness check
"Does this spec define: goal, actors, inputs, outputs, acceptance
criteria, constraints, and edge cases? What is missing?"

    The human decides which suggestions to accept or reject. The LLM proposes — the human disposes.

  3. Human approves the spec — a second human reviews it

    The spec is approved by a human who was not involved in writing it. This is the hardest gate to skip and the most valuable one. A spec that cannot be understood by a fresh reader is not ready.

  4. LLM generates tests directly from the spec

    With the approved spec, the LLM generates a comprehensive test suite. Every test must be traceable to a specific acceptance criterion. No criterion, no test.

    Prompt for test generation 
    "Given the spec below, generate a failing test for every acceptance
criterion. Label each test with the AC it covers (e.g. AC-01, AC-02).
Do not assume any implementation detail. Use [language/framework].
Every test must fail when run against an empty implementation."

[paste spec.md here]

    Human reviews every generated test. Tests that cannot be directly traced to the spec are discarded.

  5. Human runs the tests — confirms all are red 🔴

    Non-negotiable human step. Run the full test suite and confirm every new test fails. A test that passes before implementation exists is either testing the wrong thing or validating pre-existing behavior — both are dangerous assumptions that must be investigated before proceeding.

  6. LLM implements the feature against the tests

    Now the LLM generates implementation code with a precise, verifiable target.

    Prompt for implementation 
    "Implement the feature described in spec.md so that all tests in
feature.test.ts pass. Hard constraints:
- Do NOT modify any test file
- Run the FULL test suite (including pre-existing tests) before responding
- If any pre-existing test fails, fix the regression before responding
- Definition of done: all new tests GREEN + zero regressions"

[paste spec.md]
[paste feature.test.ts]
[paste relevant existing code context]

  7. Verify: new tests green 🟢 + zero regressions

    The complete test suite — new and existing — must pass. If any pre-existing test fails, the LLM must fix the regression before the PR is opened. This gate is automated via CI.

  8. Human review: spec ↔ tests ↔ code consistency

    Final human review checks that the implementation matches the spirit of the spec — not just the letter of the tests. The spec, tests, and code are committed together as a single atomic change.

LLM Role Summary by Phase

PhaseHuman doesLLM doesLLM does NOT
Spec Defines requirements, accepts/rejects suggestions, approves Finds ambiguities, suggests edge cases, checks completeness Author requirements, define behavior
Tests Reviews each test, confirms all are red, approves suite Generates test scaffolding from spec criteria Invent tests without spec basis
Code Reviews implementation, verifies regression suite Implements feature to make tests pass Modify test files, skip regression check
Why this prevents the core LLM failure mode Without STDD, an LLM asked to "implement X" will hallucinate requirements, skip edge cases, and produce code that looks correct but isn't. STDD breaks the problem into three bounded tasks — each with a verifiable output — so the LLM operates with precision at every step. It cannot drift because every output is checked against the previous phase's artifact.

Worked Example

A simple feature — user password reset via email token — implemented with STDD. Notice how each phase artifact constrains the next.

📋 spec.md — Phase 1 
# SPEC: Password Reset via Email Token
## Version: 1.0.0 | Status: Approved | Author: eng-team

## 1. Goal
Allow authenticated users to reset their password via a time-limited
email token, without exposing current credentials.

## 2. Actors
- User (unauthenticated, knows their email address)
- Email service (external, async)
- Token store (internal, Redis)

## 3. Acceptance Criteria

AC-01: Given a valid email, a reset token is generated and stored.
  - Token length: 32 bytes (hex-encoded, 64 chars)
  - Token TTL: 15 minutes
  - Only one active token per user at a time (previous tokens invalidated)

AC-02: The token is sent to the provided email address.
  - Email must match an existing account — otherwise: silent success (no enumeration)
  - Email delivery failure must not block the HTTP response

AC-03: Given a valid token, the user can set a new password.
  - Token must be consumed (deleted) after first use
  - New password: min 12 chars, at least 1 uppercase, 1 digit, 1 special char
  - On success: all existing sessions for that user are invalidated

AC-04: Expired or already-used tokens are rejected.
  - Response: 400 Bad Request (no distinction between expired/used/invalid)

AC-05: Rate limiting
  - Max 3 reset requests per email per hour
  - On limit exceeded: 429 Too Many Requests

## 4. Constraints
- Tokens must be cryptographically random (crypto.randomBytes, not Math.random)
- Token must never appear in server logs
- Endpoint must be available without authentication
🧪 password-reset.test.ts — Phase 2 
// Generated from spec.md — DO NOT modify without updating spec
// All tests must be RED before Phase 3 begins

import { describe, it, expect, beforeEach, vi } from 'vitest'
import { requestPasswordReset, consumeResetToken } from '../src/auth/password-reset'

describe('AC-01: Token generation', () => {
  it('generates a 64-char hex token for a valid email', async () => {
    const token = await requestPasswordReset('user@example.com')
    expect(token).toMatch(/^[a-f0-9]{64}$/)
  })

  it('invalidates previous tokens when a new one is requested', async () => {
    const first  = await requestPasswordReset('user@example.com')
    const second = await requestPasswordReset('user@example.com')
    const firstValid = await isTokenValid(first)
    expect(firstValid).toBe(false)
  })

  it('stores token with 15-minute TTL', async () => {
    const token = await requestPasswordReset('user@example.com')
    const ttl = await getTokenTTL(token)
    expect(ttl).toBeLessThanOrEqual(900) // 15 min in seconds
    expect(ttl).toBeGreaterThan(890)
  })
})

describe('AC-02: Email delivery', () => {
  it('returns success silently for non-existent email (no enumeration)', async () => {
    const res = await requestPasswordReset('ghost@nowhere.com')
    expect(res).toBeUndefined() // no error thrown, no token returned
  })

  it('does not block response on email delivery failure', async () => {
    mockEmailService.fail()
    const start = Date.now()
    await requestPasswordReset('user@example.com')
    expect(Date.now() - start).toBeLessThan(200) // fire-and-forget
  })
})

describe('AC-03: Token consumption', () => {
  it('allows password change with valid token', async () => {
    const token = await requestPasswordReset('user@example.com')
    await expect(consumeResetToken(token, 'NewP@ssw0rd!1')).resolves.toBeDefined()
  })

  it('deletes token after first use', async () => {
    const token = await requestPasswordReset('user@example.com')
    await consumeResetToken(token, 'NewP@ssw0rd!1')
    await expect(consumeResetToken(token, 'NewP@ssw0rd!2')).rejects.toThrow()
  })

  it('invalidates all user sessions on success', async () => {
    const sessions = await getActiveSessions('user@example.com')
    expect(sessions.length).toBe(0)
  })
})

describe('AC-04: Invalid tokens', () => {
  it('rejects expired token with 400', async () => {
    const expiredToken = await createExpiredToken()
    await expect(consumeResetToken(expiredToken, 'NewP@ssw0rd!1'))
      .rejects.toMatchObject({ status: 400 })
  })
})

describe('AC-05: Rate limiting', () => {
  it('returns 429 after 3 requests in one hour', async () => {
    for (let i = 0; i < 3; i++) await requestPasswordReset('user@example.com')
    await expect(requestPasswordReset('user@example.com'))
      .rejects.toMatchObject({ status: 429 })
  })
})
Gate check before Phase 3 Run vitest run password-reset.test.ts — every test must fail. If any pass, investigate before proceeding. Phase 3 can now begin.

STDD vs. TDD vs. SDD

STDD is not a replacement for TDD or SDD — it synthesizes and extends both by adding the missing integration between them.

Capability TDD SDD STDD
Prevents spec drift ✗ ✓ ✓
Prevents regression ✓ ✗ ✓
LLM-safe (bounded role) ~ ~ ✓
Verifiable "done" criteria ✓ ✗ ✓
Human-readable intent artifact ✗ ✓ ✓
Traceable requirement → test → code ✗ ✗ ✓
Edge-case coverage by design ~ ~ ✓
Works on brownfield codebases ✓ ~ ✓
Mandatory human review gates ~ ~ ✓
Prevents LLM hallucination in code ✗ ✗ ✓

✓ Supported  |  ~ Partially  |  ✗ Not addressed

The key insight TDD tells you when code is correct (tests pass). SDD tells you what to build (the spec). STDD is the only methodology that explicitly connects the two — tests are not written from memory or convention; they are derived directly and traceably from the spec.

Why STDD Matters Now

AI coding tools have fundamentally changed the cost structure of software development. Writing code is no longer the expensive step — deciding what to write and verifying it is correct are. STDD is designed for exactly this reality.

🛡️

No Silent Regressions

The full regression suite runs before any code is merged. LLM-generated code that breaks existing contracts is caught immediately, not in production.

🎯

Unambiguous "Done"

A feature is complete when all spec-derived tests are green. There is no subjectivity. Reviews become much faster because the criteria are explicit.

📎

Full Traceability

Every test maps to an acceptance criterion. Every line of production code exists to make a test pass. Audits, postmortems, and onboarding all benefit.

🤖

LLM as Amplifier, Not Author

The LLM never decides what to build. It executes well-defined tasks: generate tests from spec, generate code from tests. Human judgment stays in control.

🔄

Safe Iteration

Changing a feature means updating the spec first, then the tests, then the code — in that order. Scope creep and design drift are structurally prevented.

📦

Works on Legacy Codebases

Unlike spec-as-source approaches, STDD requires no rewrite of existing systems. It can be adopted incrementally, one feature at a time.

Known Trade-offs

STDD is not free. It requires more upfront discipline than simply prompting an LLM and shipping whatever comes out. Teams adopting STDD should expect:

  • Higher initial cost per feature — Writing a thorough spec and reviewing LLM-generated tests takes real time. The payoff is in reduced debugging, fewer incidents, and faster reviews.
  • Spec quality determines everything — A vague spec produces vague tests and unpredictable code. The methodology raises the stakes on specification quality.
  • Not ideal for exploratory work — Prototypes, spikes, and discovery work should not use STDD. Apply it when building features for production systems with real users.
  • Requires test discipline — STDD only works if the team is committed to never modifying tests to make them pass. This requires cultural buy-in, not just tooling.
"The spec is the most important artifact. The tests are its executable form. The code is just the cheapest thing that makes the tests pass." — STDD Manifesto

Getting Started

You do not need new tools to adopt STDD. Start with:

1. Pick one upcoming feature
2. Write spec.md before opening your editor
3. Get spec approved by a teammate
4. Prompt your LLM to generate tests from the spec
5. Confirm all tests are red
6. Prompt your LLM to implement the feature against the tests
7. Verify: new tests green, no regressions
8. Commit spec + tests + code as one atomic PR

After three features, the pattern will feel natural. After ten, you will not want to develop any other way.
